# Story 10.3: LLM-Enriched Daily Greeting Line

Status: done

## Story

As Laurus,
I want the daily view greeting line to be dynamically generated by the on-device LLM when the model is available — reflecting my actual relationship state at that moment,
So that opening Spetaka feels genuinely personal rather than cycling through a static rotation.

## Acceptance Criteria

### AC1 — Static fallback shown immediately, no delay

**Given** `DailyViewScreen` loads (or re-enters foreground)
**When** the `GreetingLineWidget` renders
**Then** it immediately displays a static fallback greeting from a hardcoded pool in `GreetingService.staticFallback(urgentCount, concernCount)` — no loading spinner, no blank space, no delay; the UI is immediately complete

### AC2 — LLM greeting generated in background when model ready

**Given** `ModelManager.isModelReady = true`
**When** the static greeting is shown
**Then** `GreetingLineNotifier` triggers LLM generation in the background (non-blocking, no `await` on the render path):
  - Context passed to the prompt: urgent friend count, concern flag count
  - Prompt template from `PromptTemplates.greetingLine(urgentCount: ..., concernCount: ..., userName: 'Laurus')`
  - Calls `LlmInferenceService.infer(prompt)` (30s timeout enforced by `LlmInferenceService`)
  - On response: `greetingLineProvider` state is updated with the LLM result
  - `GreetingLineWidget` reacts via Riverpod and smoothly cross-fades (300ms `easeInOutCubic`) from the static greeting to the LLM greeting

### AC3 — Static retained when model not ready or timeout

**Given** `ModelManager.isModelReady = false` OR `LlmInferenceService.infer()` returns empty list (timeout)
**Then** the static greeting remains — no update, no error shown; the experience is identical to the pre-LLM version

### AC4 — Static pool ≥ 8 variants, coach tone

**Given** `GreetingService.staticFallback()` is called
**Then** the static pool contains at minimum 8 greeting variants covering:
  - 0 urgent + 0 concern
  - 1 urgent + 0 concern
  - 2+ urgent + 0 concern
  - concern-flagged friend(s) present (regardless of urgentCount)
**And** all variants are in coach tone, never metric-framed (no counts or scores mentioned)

### AC5 — `greetingLineProvider` always returns a non-null String

**Given** `greetingLineProvider`
**Then** it is a `@riverpod class GreetingLineNotifier` returning `String` (never null/async from the widget's perspective — always has a valid value); the static fallback is the initial value set in `build()`

### AC6 — Cross-fade at 60fps, height stable, TalkBack correct

**Given** the cross-fade animation between static and LLM greetings
**Then** it runs at 60fps using `AnimatedSwitcher` with `FadeTransition` child — no `AnimatedContainer` resize (greeting line height stays constant via `Stack`-based `layoutBuilder`)
**And** TalkBack reads the most recent greeting text; transition does not cause focus jump (NFR17)

## Dev Notes

### Architecture Constraints (MUST FOLLOW)

- **Provider code-gen:** Always `@riverpod` annotation + `build_runner` — never write manual providers. [Source: architecture.md#Riverpod Patterns]
- **Prompt centralization:** ALL prompt strings in `PromptTemplates` — never construct prompts inline. [Source: architecture-phase2-addendum.md#Enforcement Guidelines]
- **Air-gapped inference:** `LlmInferenceService.infer()` makes zero network calls (NFR19). Only the model download uses the network.
- **Non-blocking:** LLM call must never block the render path. Use `Future.microtask()` in `build()` to defer to next tick. [Source: architecture-phase2-addendum.md#GreetingService]
- **Singleton LLM service:** `llmInferenceServiceProvider` is `keepAlive: true` — reuse the cached instance, never instantiate directly.
- **Theme tokens:** Use `Theme.of(context).colorScheme.*` — never hard-code hex colors. Tokens at `lib/shared/theme/app_tokens.dart`. [Source: Story 4.7 dev notes]
- **Localization:** The greeting text itself is dynamically generated (LLM or static pool) — not a localization key. No `context.l10n.*` needed for the greeting string. New UI labels (if any) use `context.l10n.*`.
- **Logging:** `dart:developer log()` — never `print()`. [Source: architecture.md#Key Rules]
- **`GreetingLineNotifier` auto-dispose:** Do NOT add `keepAlive: true`. The greeting should refresh on every daily view visit.

### Anti-Patterns (FORBIDDEN)

- ❌ Showing a loading spinner or blank space for the greeting line — **AC1 requires instant static fallback**
- ❌ `await`-ing the LLM call on the render path (blocks UI thread)
- ❌ Constructing prompt strings inline outside `PromptTemplates` — architecture enforcement
- ❌ Persisting LLM greeting to SQLite — session-only, in-memory Riverpod state only
- ❌ Calling `LlmInferenceService` from `GreetingService` directly — `GreetingService` stays pure Dart; the Riverpod notifier orchestrates the LLM call
- ❌ Using `AnimatedContainer` for the greeting transition — use `AnimatedSwitcher` + `FadeTransition` per AC6
- ❌ Removing `Key('greeting_banner')` from the widget tree — existing widget tests find by this key
- ❌ Changing the existing `GreetingService.greeting()` signature — 35 tests depend on it

### Key Implementation Details

#### `GreetingService.staticFallback()` — 8-variant pool

The method must select from 2 variants per scenario (4 scenarios × 2 = 8 minimum) using `DateTime.now().minute % 2` for variety:

| Scenario | Variants |
|---|---|
| `concernCount > 0` | "quelqu'un vous tient à cœur..." / "la bienveillance commence par un signe simple..." |
| `urgentCount == 0` | "tout est calme, c'est le bon moment pour souffler 🌿" / "une journée tranquille s'ouvre devant vous 🌿" |
| `urgentCount == 1` | "une belle occasion de renouer ce lien aujourd'hui ✨" / "un seul geste peut faire toute la différence ✨" |
| `urgentCount >= 2` | "quelques liens vous attendent, allez-y à votre rythme 🌟" / "plusieurs connexions s'offrent à vous 🌟" |

All variants must include `userName` prefixed by the time salutation (e.g., "Bonjour, Laurus — ...").

Priority: concern check before urgentCount (a concerned friend is always the priority framing).

#### `PromptTemplates.greetingLine()` — French LLM prompt

```dart
static String greetingLine({
  required String userName,
  required int urgentCount,
  required int concernCount,
}) {
  final contextHint = concernCount > 0
      ? "Certains de ses proches méritent une attention particulière aujourd'hui."
      : urgentCount == 0
          ? "Tout est calme dans ses relations aujourd'hui."
          : urgentCount == 1
              ? "Un de ses proches a besoin d'un signe aujourd'hui."
              : "Plusieurs de ses proches méritent un geste aujourd'hui.";

  return '''Tu es un coach relationnel bienveillant. Génère UNE SEULE ligne de salutation chaleureuse pour accueillir $userName.
$contextHint
Règles strictes :
- Maximum 15 mots au total
- Ton encourageant, jamais culpabilisant
- Langue : français uniquement
- Aucun chiffre ni métrique
- Aucun guillemet autour de ta réponse
- Une seule phrase courte, directe''';
}
```

#### `GreetingLineNotifier` — `_llmFired` guard pattern

```dart
@riverpod
class GreetingLineNotifier extends _$GreetingLineNotifier {
  bool _llmFired = false;

  @override
  String build() {
    final dailyAsync = ref.watch(watchDailyViewProvider);
    final entries = dailyAsync.valueOrNull ?? [];

    final urgentCount = entries
        .where((e) => e.prioritized.tier == UrgencyTier.urgent)
        .length;
    final concernCount = entries
        .where((e) => e.friend.isConcernActive)
        .length;

    // Fire LLM exactly once per provider lifetime (per daily view visit)
    if (!_llmFired && dailyAsync is AsyncData) {
      _llmFired = true;
      Future.microtask(
        () => _generateAsync(urgentCount: urgentCount, concernCount: concernCount),
      );
    }

    return GreetingService().staticFallback(
      urgentCount: urgentCount,
      concernCount: concernCount,
    );
  }

  Future<void> _generateAsync({
    required int urgentCount,
    required int concernCount,
  }) async {
    try {
      final modelState = ref.read(modelManagerProvider);
      if (modelState is! ModelReady) return;

      final prompt = PromptTemplates.greetingLine(
        userName: 'Laurus',
        urgentCount: urgentCount,
        concernCount: concernCount,
      );

      final results = await ref.read(llmInferenceServiceProvider).infer(prompt);
      if (results.isNotEmpty && results.first.trim().isNotEmpty) {
        state = results.first.trim();
      }
    } catch (e) {
      dev.log('GreetingLineNotifier: LLM error — $e', name: 'ai.greeting');
    }
  }
}
```

#### `GreetingLineWidget` — AnimatedSwitcher + height stability

The `ValueKey<String>(greeting)` on the `Text` child is **critical** — it tells `AnimatedSwitcher` when the content has changed so it triggers the cross-fade animation. Without it, swapping the same widget class/type is not detected.

The `layoutBuilder` using `Stack` is essential to prevent height change. Without it, `AnimatedSwitcher` uses `Column`-like layout that resizes when the new child has different height.

The outer `Key('greeting_banner')` (set on the `Padding` or `KeyedSubtree`) must be preserved for existing widget tests to find the banner.

#### `DailyViewScreen` changes — audit of `hasConcern` usage

Current `_DailyList.build()` computes `hasConcern` ONLY for the greeting. It is **not** passed to `HeartBriefingWidget` (which takes `entries` directly). Safe to remove.

The `greeting` local variable is also only used for `_GreetingBanner`. Safe to remove.

After removal, add `const` to `SliverToBoxAdapter(child: GreetingLineWidget())` — it is now a `const` constructor.
